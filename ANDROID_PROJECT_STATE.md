## SECTION 1 — APPLICATION OVERVIEW

### What is Jarvis Collector?

**JARVIS Collector** is an Android application that acts as three things simultaneously:

1. **Signal Collector** — captures SMS messages from the device and queues them for upload to Supabase.
2. **Thin Client** — downloads structured insights generated by the remote JARVIS AI pipeline (Facts, Tasks, FYI events, Financial events, Daily Briefs) and caches them locally in Room.
3. **Notification Surface** — displays prioritised notifications, morning briefs, and reminders, routing deep links to specific screens.

> **Architectural Law**: The Android app performs **zero AI reasoning**. No classification, no LLM calls, no financial computations that determine facts. All intelligence originates on Supabase / the backend pipeline. Android only reads, caches, and surfaces data.

### Core Feature Areas

| Feature | Purpose | Data Source |
| :--- | :--- | :--- |
| **Tasks** | Actionable todo items assigned by Supabase agents | `todo_items` → Room `todos` |
| **Facts** | Long-term memory facts about the user | `facts` → Room `facts` |
| **FYI** | Short-form category alerts (Family, Health, Shopping, Travel) | `fyi_events` → Room `fyi_events` |
| **Financial** | Transaction ledger, spending aggregates, category breakdown, monthly trends | `financial_events` → Room `financial_events` |
| **Daily Briefs** | AI-generated morning briefings | `daily_briefs` → Room `daily_briefs` |
| **Notifications** | Priority notification items | Generated locally from signals / insights |

## SECTION 2 — HIGH LEVEL ARCHITECTURE

```
┌─────────────────────────────────────────────────────────────────────┐
│                     SUPABASE (jarvis_insights_schema)               │
│  todo_items  |  facts  |  fyi_events  |  financial_events           │
│  financial_facts  |  daily_briefs  |  user_preferences              │
└────────────────────────────┬────────────────────────────────────────┘
                             │  REST GET  (Accept-Profile: jarvis_insights_schema)
                             │  REST PATCH (Content-Profile: jarvis_insights_schema)
                             ▼
┌─────────────────────────────────────────────────────────────────────┐
│  InsightSyncService.kt  (Kotlin object, called from WorkManager)    │
│  – Fetches all tables in parallel                                   │
│  – Maps JSON → Room entities inside a single db.withTransaction {}  │
│  – Writes sync diagnostics per entity to sync_diagnostics table     │
└────────────────────────────┬────────────────────────────────────────┘
                             │ Room (jarvis_mobile.db, version 5)
                             ▼
┌─────────────────────────────────────────────────────────────────────┐
│  JarvisDatabase  (Room singleton)                                   │
│  11 entities · 11 DAOs                                              │
└────────────────────────────┬────────────────────────────────────────┘
                             │ suspend fun / Flow<T>
                             ▼
┌─────────────────────────────────────────────────────────────────────┐
│  Repository objects  (Kotlin objects, not classes)                  │
│  – Read from DAO                                                    │
│  – Write to Supabase first (Remote-First Law)                       │
│  – Write to Room only on Supabase success                           │
└────────────────────────────┬────────────────────────────────────────┘
                             │ StateFlow / Flow
                             ▼
┌─────────────────────────────────────────────────────────────────────┐
│  ViewModels  (AndroidViewModel subclasses)                          │
│  – Expose StateFlow<UiState>                                        │
│  – Auto-refresh via Flow.collectLatest                              │
└────────────────────────────┬────────────────────────────────────────┘
                             │ .collectAsState()
                             ▼
┌─────────────────────────────────────────────────────────────────────┐
│  Compose UI Screens                                                 │
│  – Dark slate theme (#0F172A background, #1E293B cards)             │
│  – Jetpack Navigation Compose                                       │
└─────────────────────────────────────────────────────────────────────┘
```

### State Management Pattern
- ViewModels expose `StateFlow<UiState>`.
- `UiState` is a `data class` with loading/error flags.
- Compose screens observe via `.collectAsState()`.
- Live Room updates use `Flow<List<T>>` from DAOs triggering `.collectLatest {}` in ViewModel `init{}`.

### Synchronization Flow
- `InsightSyncWorker` (WorkManager) calls `InsightSyncService.syncInsights(context)` on a periodic schedule.
- All table fetches are done via `JarvisInsightsClient.fetchTable(tableName)` using OkHttp3 with schema header `Accept-Profile: jarvis_insights_schema`.
- Each entity sync is wrapped in its own try/catch, writing to `sync_diagnostics` on success or failure.

## SECTION 5 — SUPABASE SYNC INVENTORY

All syncs use schema `jarvis_insights_schema` via `Accept-Profile` / `Content-Profile` headers.

| Entity | Supabase Table | Android Table | Key Field Mappings | Status |
| :--- | :--- | :--- | :--- | :--- |
| Todos | `todo_items` | `todos` | `todo_id→todo_id`, `source_reference→source_signal_id` | ✅ WORKING |
| Facts | `facts` | `facts` | `fact_id or id→id`, `fact_type→title`, `fact_value→summary`, `source_agent→source` | ✅ WORKING |
| FYI Events | `fyi_events` | `fyi_events` | `event_id or fyi_event_id→fyi_event_id`, `description→summary` | ✅ WORKING |
| Financial Events | `financial_events` | `financial_events` | `id→financial_event_id`, `title→merchant`, `event_date→event_timestamp` | ✅ WORKING |
| Financial Insights | `financial_facts` | `financial_insights` | `merchant_canonical→title`, `fact_type→type`, `event_date→dueDate` | ✅ WORKING |
| Daily Briefs | `daily_briefs` | `daily_briefs` | `brief_id or id→id`, `content or items_json→itemsJson`, `generated_at→generatedAt` | ✅ WORKING |
| User Preferences | `user_preferences` | `user_preferences` | Direct 1:1 mapping | ✅ WORKING |
| Notifications | N/A (local) | `notifications` | Generated locally | ⚠️ LOCAL ONLY |

### Critical Known Constraints
1. **`facts` table on Supabase has NO `read_flag` column**. Never include `read_flag` in a PATCH payload to `facts`. The field exists in Room only.
2. **Todo updates** must target table `todo_items`, NOT `todos`.
3. **Financial insight updates** must target table `financial_facts`, NOT `financial_insights`.

### Architecture in One Paragraph
JARVIS Android is a thin Compose client backed by a Room database (`jarvis_mobile.db`, v5). All data originates from Supabase (`jarvis_insights_schema` schema). A WorkManager periodic job calls `InsightSyncService.syncInsights()` which fetches all tables via `JarvisInsightsClient` (OkHttp3, REST) and upserts them into Room inside a transaction. ViewModels observe Room via `StateFlow`, and screens use `.collectAsState()`. All writes to Supabase use `JarvisInsightsClient.updateRow()` before updating Room (Remote-First law).

### Critical Constraints — Read These Before Writing Any Code

| Constraint | Detail |
| :--- | :--- |
| **Supabase schema header** | All REST calls must include `Accept-Profile: jarvis_insights_schema` |
| **facts table** | Has NO `read_flag` column on Supabase — never PATCH this field |
| **todo updates** | Target `todo_items` table on Supabase (NOT `todos`) |
| **financial_insights updates** | Target `financial_facts` table on Supabase (NOT `financial_insights`) |
| **Remote-First** | Update Supabase → success → update Room. If Supabase fails, Room stays unchanged |
| **No intelligence on Android** | Android never classifies, generates, or reasons. It reads Supabase output |
| **Income heuristic** | In `FinancialDashboardViewModel`, income = title contains "received/refund/credit/deposit/income/salary" |
